Flask源码之请求上下文和应用上下文

发表于 2020-12-24 更新于 2020-12-29 分类于 flask

我们继续来看wsgi这个方法，self.request_context这个方法把environ封装成了RequestContext对象，看名字也知道这个是请求上下文

简单看一下RequestContext的定义，位置在ctx.py中

我们看到， 第一个参数是Flask对象，所以传入了self（注意request_context方法是Flask类的方法，所以self就是Flask的实例）

我们还看到RequestContext有request这个属性

回到wsgi_app这个方法，我们看到接下来ctx.push这一句调用了RequestContext的push方法，这个方法就是把自身也就是 RequestContext压入到 LocalStack这个数据结构中

Local和LocalStack

flask中实现了Local和LocalStack两种数据结构，Local的作用是提供线程隔离和协程隔离，类似threadlocal

12345是线程id，RequestContext是请求上下文，这样做是为了在wsgi server是多线程或多协程的情况下，每个线程或协程获取自己的上下文

但实际上，flask是把RequestContext存储在栈（后进先出）结构中，类似这样

这样做是因为一个请求可能有多个请求上下文，比如服务端重定向的情况（一个请求A视图，我们需要重定向到B视图，这就需要自己再push一个RequestContext）

此外还有多个应用上下文的情况

接下来我们再看push方法

首先从_request_ctx_stack的栈顶获取一个请求RequestContext

_request_ctx_stack的定义再globals.py中

它现在还是的状态是这样，栈是空的

继续看push方法，如果栈是空的话，我们把self也就是RequestContext压入栈，在最后一句，注意push是RequestContext的方法

我们还注意到这里还有一个_app_ctx_stack，这也是LocalStack，位置在globals.py

只不过这个栈里面存储的是应用上下文，也就是说

然后我们也执行了app_ctx.push()方法，也就是把应用上下文压入栈

到这里我们就知道了，执行RequestContext对象的push方法会把自身压入_request_ctx_stack中，还会把AppContext也压入_app_ctx_stack中

如何获取请求上下文

我们常常需要在一个视图函数中获取请求中的参数，例如url，remote_address

我们当然可以每次手动获取栈顶的RequestContext对象，然后调用RequestContext的request属性，但每次操作栈结构还是有点繁琐

flask的做法是使用LocalProxy

LocalProxy

request就是LocalProxy，它是代理模式在flask中的运用，作用就是代理对RequestContext.request的操作

例如，我们使用request.remote_addr实际上会先从栈顶获取RequestContext然后获取request属性。

我们执行request.remote_addr就相当于执行 _request_ctx_stack.top.request.remote_addr

难为什么用LocalProxy而不是直接request=_request_ctx_stack.top.request呢

原因是这样写，项目启动的时候，这句就执行了，就直接从栈顶获取RequestContext对象了，这时候还没有呢。

我们看看LocalProxy是怎么做的

再看LocalProxy

  1. __getattr__

    我们request是一个LocalProxy对象，那么request.remote_addr会执行 __getattr__ 这个方法，这个方法又会去_get_current_object获取被代理的对象，也就是 RequestContext.request，之后，调用 getattr(self._get_current_object(), name)获取 RequestContext.request的指定属性

  2. _get_current_object

    这个方法执行self.__local()，获取被代理对象

  3. __init__

    在__init__方法中我们可以看到object.__setattr__(self, "_LocalProxy__local", local)，__local也就是传入进来 `partial(_lookup_req_object, “request”)``

    所以self.__local()也就是执行偏函数 partial(_lookup_req_object, "request")，也就是 _lookup_req_object("request")

总结

到这里，我们就可以看出flask中的请求上下文是如何存储，获取的

请求上下文存储在LocalStack结构中，Local是为了线程安全，Stack是为了多请求上下文的场景

而获取是通过LocalProxy，使用代理模式是为了动态地获取请求上下文，在访问request属性的时候，才会从栈中获取真实的请求上下文，然后代理属性的获取

flask中还有应用上下文，curren_app，我们有时候会通过current_app.config来获取配置信息，原理和request类似，只不过代理的是AppContext对象

Flask源码之从WSGI协议说起

发表于 2020-12-22 更新于 2020-12-23 分类于 flask

引言

我们知道web应用的本质就是：

  1. 浏览器发送一个HTTP请求
  2. 服务器收到请求，处理业务逻辑，生成html、json等数据
  3. 服务器把html、json等数据放在HTTP响应的body中发送给浏览器
  4. 浏览器收到http响应

可以看到这一过程我们需要接受、解析HTTP请求和发送HTTP响应，如果这些都由我们自己来写的话，我们需要自己处理包括建立TCP连接（HTTP协议是建立在TCP之上）、解析原始HTTP请求等工作，这太麻烦了。所以我们需要：

  1. 一个HTTP服务器软件帮我们处理这些工作
  2. Web应用框架处理业务逻辑

而WSGI就是约定HTTP服务器软件和Web应用框架交互的协议

安装python开发环境包括安装python,pip,pycharm,搜狗输入法

安装python,pip

ubuntu18.04内置了python3.6.9

但是没有安装pip

有了pip3，接下来用pycharm create new project就不会出现ModuleNotFoundError: No module named 'distutils.util'错误了

  • pip和pip3的区别

python 有python2和python3的区别
那么pip也有pip和pip3的区别
大概是这样的
1、pip是python的包管理工具，pip和pip3版本不同，都位于Scripts\目录下：
2、如果系统中只安装了Python2，那么就只能使用pip。
3、如果系统中只安装了Python3，那么既可以使用pip也可以使用pip3，二者是等价的。
4、如果系统中同时安装了Python2和Python3，则pip默认给Python2用，pip3指定给Python3用。
5、重要：虚拟环境中，若只存在一个python版本，可以认为在用系统中pip和pip3命令都是相同的
